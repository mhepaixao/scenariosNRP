Abstract

The next release problem is a significant task in the iterative and incremental software development model, involving the selection of a set of requirements to be included in the next software release. Given the dynamic environment in which modern software development occurs, the uncertainties related to the input variables considered in this problem should be taken into account. In this context, this paper proposes a novel formulation to the next release problem based on scenarios and considering the robust optimization framework, which enables the production of robust solutions. In order to measure the “price of robustness”, several experiments were designed and executed over artificial and real-world instances. All experimental results are consistent to show that the penalization, in terms of solution quality, due to robustness is significantly small, which qualifies the proposed model to be applied even in large-scale real-world software projects.

Instances

Synthetic Instances

The artificial instance set is composed by 7 randomly generated instances. Each instance has 3 scenarios and the requirement value can assume an integer between 1 and 10. The instances were generated with different numbers of requirements, ranging from 50 to 200. Different approaches to the cost standard deviation were considered. Each instance have its standard deviation adjusted to 10%, 20%, 30%, 40% and 50% of the respective cost. A random approach was also applied, where the standard deviation is set as a random number between 0 (exclusive) and 50% of the cost. The artificial instance name is in the format I_S_R_D, where R is the number of requirements and D is the standard deviation approach. The instance I_S_100_20, for example, has 100 requirements and the standard deviation is set to 20% of the requirement cost.

Real-world Instances

The real instances were extracted from bug repositories of two big open source projects, Eclipse (a java integrated development environment) and Mozilla (a set of web applications). This strategy for NRP real instance extraction was adapted from the work by Xuan et al. A bug repository is a forum where users (end users, developers, testers, etc) can report bugs related to the project. In this forum, one bug report may be commented by many users. One bug report is mapped to a requirement. The requirement value of a bug is then calculated as the number of users that commented on that particular bug. In addition, the bug severity is mapped to the requirement cost. Both the requirements value and cost are normalized to the 1 to 10 interval. In all cases, each instance was considered to have only one scenario (with a probability equals to 1). Seven instances, composed by the most important requirements, were extracted from each bug repository. The instances contains 50, 70, 100, 120, 150, 170 and 200 requirements, respectively. Likely the synthetical instances, each real instance have its standard deviation adjusted to 10%, 20%, 30%, 40% e 50% of the respective cost. The random approach was also considered. The real-world instance name is in the format I_Re_P_R_D, where P represents the project (E for Eclipse and M for Mozilla), R is the number of requirements, and D is the standard deviation approach. The instance I_Re_E_120_30, for example, was generated from the eclipse bug repository, has 120 requirements and the standard deviation set to 30% of the cost.
